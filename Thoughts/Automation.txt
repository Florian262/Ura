2. Tool 1: Automating localhost Visual & UI Testing (Playwright)
Instead of manually opening your browser, navigating to localhost:5173, and clicking through tabs to see if the level selector or chapter engine works, you use Playwright. It launches a headless, local browser instance that acts exactly like a human user.

How to Automate It
Install it locally: npm i -D @playwright/test

Create a local testing script (tests/chapterEngine.spec.ts):

TypeScript
import { test, expect } from '@playwright/test';

test('Should load level dashboard and navigate to A1 Chapter 1 seamlessly', async ({ page }) => {
  // 1. Tell the automation tool to open your running local development server
  await page.goto('http://localhost:5173/');

  // 2. Click the primary CTA button on your landing page
  await page.click('text=FILLO MËSIMIN');

  // 3. Verify it routed successfully to the Level Dashboard
  await expect(page).toHaveURL(/.*dashboard/);

  // 4. Click the active A1 Level card
  await page.click('text=01. NIVELI A1');
  
  // 5. Verify the sticky top navigation panel anchors exist
  const stickyHeader = page.locator('nav');
  await expect(stickyHeader).toBeVisible();
});
3. Tool 2: Automating Text & Translation Double-Checks (Vale)
To double-check the text written inside your Turkish-Albanian database or local JSON files for typos, casing, or formatting issues, you can use an automated command-line tool called Vale.

How to Automate It
Vale is a syntax-aware linter for prose. You point it at your src/infrastructure/mock/a1_chapter1_data.ts or markdown curriculum files.

You can create custom local linting rules to make sure:

Turkish words always use the correct unique characters (ç, ğ, ı, ö, ş, ü).

Albanian translation strings always follow clean rules (checking for proper placement of ë and ç).

Running vale src/ instantly lists every single typo or structural text mistake in your terminal before you compile the app build.

4. Tool 3: Automating the Core Engine Validation (Vitest)
Your vowel harmony and suffix mutation rules must be guarded by automated unit tests. If you accidentally break an A1 plural rule while adjusting an engineering logic parameter later, your pipeline will instantly catch it.

How to Automate It
Using Vitest, you write short validation checks:

TypeScript
import { describe, it, expect } from 'vitest';
import { pluralStrategy } from '../src/core/grammar/strategies/PluralStrategy';

describe('Turkish Plural Suffix Automation Rules', () => {
  it('should correctly apply -lar for back vowels (Kalın)', () => {
    expect(pluralStrategy('Okul')).toBe('Okullar'); // School -> Schools
    expect(pluralStrategy('Dollap')).toBe('Dollaplar'); 
  });

  it('should correctly apply -ler for front vowels (İnce)', () => {
    expect(pluralStrategy('Ev')).toBe('Evler'); // House -> Houses
    expect(pluralStrategy('Xham')).toBe('Xhamlar'); 
  });
});
5. How to Make Your Pipeline Efficient (Husky Git Hooks)
To make this completely hands-off and automated, you use Husky (npm i -D husky). This links your automation tools directly to your code save repository.

Every time you type git commit -m "added a1 chapter 2", Husky intercepts the command and automatically fires your tools in the background:

Bash
# What happens automatically behind the scenes on your local machine:
npm run lint      # Checks your code for TypeScript syntax alignment
vale src/         # Scans all written text and translations for typos
vitest run        # Runs all core Turkish grammar rule strategy math tests
npx playwright test # Opens localhost headlessly, clicks all tabs, checks for runtime crashes
If a single translation is misspelled or a button link breaks on localhost, the pipeline halts the commit, flags the exact error file line, and protects your production build from breaking. If everything passes, your build is verified as perfect automatically!
